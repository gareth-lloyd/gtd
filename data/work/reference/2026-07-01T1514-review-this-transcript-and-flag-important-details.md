  ### High-level summary of the proposal
  - **Problem:** connecting a new PMS-integrated property is fully manual. IMs hand-set per-hotel
    Check-in + Upsell/add-on config flags (often via scripts that only *look* automated). Error-
    prone, generates support tickets, doesn't scale.
  - **Two options weighed** (Decision Brief): A = derive config flags from PMS capabilities live;
    B = versioned per-vendor config templates applied by Canary.
  - **Tension:** the meeting verbally leaned Option A (nullable/read-only flags, config-as-
    computed-property-from-capabilities). But the code-grounded brief showed pure A is unsafe
    today — e.g. CloudBeds hotels have nationality/marketing_consent ON yet ZERO carry those
    capabilities (deriving would silently flip ~226 nationality / ~194 marketing-consent OFF), and
    integration_precheckin_method (prereg/notes) has no capability to derive from.
  - **Shipped v1 design = effectively Option B (hybrid):** a Canary-side versioned template
    registry keyed by Vendor. Cloud Beds only, ~14 hardcoded fields (Check-in + Upsells), behind
    GrowthBook flag `pms-auto-config`, per-hotel. Gateway publishes a new `pms-config-change-
    requested` Kafka event (replacing/demoting the 3-hour polling cron) → new Canary Kafka
    consumer takes a per-hotel Redis lock, re-reads LIVE gateway state, refreshes mirrors, applies
    template in one txn, emits `pms-config-applied`. Audit row per run on Manage Properties feed,
    attributed to the connecting admin. Pre-existing hotels recorded + left untouched. Per-PMS
    default overrides live in the Gateway; per-hotel overrides use existing (now nullable/read-
    only) config fields. Add-on/upsell PRICING exposure explicitly out of scope.
