  Strong, well-reasoned PRD. The core design (Blake's room↔device binding: device proves

  ### Code-validated finding: the "build a net-new JWT system" framing is wrong — reuse is stronger than the PRD suggests
  The PRD flags the existing Kiosk device-pairing model as prior art "for eng to weigh."
  I checked the code. The reuse case is stronger than the PRD implies, AND the PRD has one
  factual error about it:

  - **The kiosk device token is an opaque DB UUID, not a JWT.** `HotelDevice.token`
    (backend/canary/hotels/models/hotel_device.py) is a `uuid4()` persisted on the row;
    validation is a direct DB lookup (`HotelDevice.objects.get(token=...)`), not signature
    verification. (There is a separate `kiosk_session` JWT that *embeds* the device token
    for WebSocket auth — that's likely where the "JWT" impression came from.)
  - **This matters:** an opaque DB token gives revocation *for free* — kill the row / flip
    status and the token is dead instantly, no deploy. The PRD's own component #1 asks for
    a `jwt_id (for revocation)` and component #7 for revoke/re-bind — both are native to the
    opaque-token model and *harder* with a stateless JWT (you'd need a revocation list to
    undo a signed token's durability). So the PRD's stated requirements actually argue
    against the JWT it proposes. Recommend defaulting to extending the opaque-token primitive
    unless there's a concrete reason for self-validating tokens.
  - **Reuse is real but not plug-and-play (~1–2 wks).** `HotelDevice` is hotel-scoped only —
    no room field, no reservation FK. To reuse you'd add `room`/`room_number` + a
    `unique(hotel, room)` constraint, and write the "current checked-in reservation for this
    room" resolver. Frontend persistence (localStorage + 5yr cookie, `useHotelDeviceStore`)
    and the fanout/WebSocket scope framework are reusable as-is. Coupling to check-in
    *session* flows is looser than feared — room devices can be added without refactoring the
    session model. Per the PRD's own note: loop in KSK (Navid) early since the primitive is
    their territory.

  ### Risks / gaps to raise at the walkthrough
  1. **Occupancy resolver is the real hard part, not the binding.** "Current checked-in
     reservation for room X" via `room_number` string matching is fragile: room moves,
     suite/multi-res rooms, day-use overlaps, and the check-in vs checkout race all hit here.
     The binding mechanism is easy; the resolver semantics deserve the design time. Confirm
     the authoritative source (CheckIn status vs Reservation flags) and how concurrent /
     overlapping reservations on one room_number resolve.
  2. **Polling latency at the turnover boundary.** ~30s poll means a ~30s window where a
     departed guest's Compendium is still shown post-checkout, and a just-arrived guest sees
     generic mode. Success criterion says "within one poll cycle" — confirm 30s exposure of
     prior-guest data at checkout is acceptable to Crown/security, or shorten the cycle on the
     checkout transition.
  3. **MDM is load-bearing and unconfirmed (Open Q1).** The entire security model rests on
     kiosk mode preventing token extraction, yet MDM posture at Crown beta rooms is still
     open. This should be answered *before* build kickoff, not during — it's a go/no-go input,
     not a detail. If Crown can't guarantee MDM, the fallback (short-lived rotating tokens)
     materially changes scope.
  4. **Device fingerprint binding (Open Q7) downgrades the MDM dependency.** Binding the token
     to a device fingerprint makes a copied token useless off-device — that's the cleanest
     defense against the replay-after-checkout threat and reduces reliance on MDM. Worth
     pulling into v1 rather than fast-follow, especially if MDM can't be guaranteed.
  5. **Customer-commitment framing.** Belinda confirmed in-thread this is part of Crown's
     Compendium rollout commitment (no timeline committed, Crown knows it's WIP). Decoupled
     from the IoT vendor decision (Honeywell/Creston → Interel) — good, keep it that way.
